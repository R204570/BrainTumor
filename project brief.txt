## WHAT I HAVE BUILT (context)

I have trained a 3D U-Net CNN model on the BraTS dataset for brain 
tumor segmentation. The model takes an MRI scan as input and outputs 
the following structured data per patient session:

  wt_volume_cm3    — whole tumor volume
  ncr_volume_cm3   — necrotic core volume
  ed_volume_cm3    — peritumoral edema volume
  et_volume_cm3    — enhancing tumor volume
  diameter_mm      — max tumor diameter
  centroid_x/y/z   — tumor centroid in voxel space
  lobe_frontal     — bool: frontal lobe involved
  lobe_temporal    — bool: temporal lobe involved
  lobe_parietal    — bool: parietal lobe involved
  lobe_occipital   — bool: occipital lobe involved
  lobe_other       — bool: other region involved

The model also produces annotated MRI slice images showing 
segmentation overlays (color-coded ET, NCR, ED regions).

I have also designed and documented:
  - A full textual input schema (symptoms, genomics, VASARI features,
    clinical metrics, lab/pathology data, treatment history)
  - A complete system prompt for Llama-3.3-70b (via Groq API)
  - A PostgreSQL DDL with 7 tables: patients, sessions,
    patient_context (JSONB), messages, questions_asked,
    imaging_reports, diagnostic_reports, knowledge_chunks (pgvector)
  - A WHO 2021 grading rule engine (5 decision rules)
  - A survival scoring algorithm (0–100 point system)
  - A JSON schema for two LLM response types:
      {"response_type": "question", ...}  — MCQ or MSQ question
      {"response_type": "report", ...}    — WHO grade + survival report
  - The full RAG request loop logic

I am attaching:
  [1] neuroassist_spec.html  — full system prompt + JSON schemas + 
      PostgreSQL DDL (all documented in tabbed format)
  [2] architecture.svg       — system architecture diagram showing:
      U-Net outputs + Clinician UI → PostgreSQL context store →
      RAG retriever + Field completeness tracker →
      Llama-3.3-70b agent → MCQ/MSQ question OR diagnostic report →
      feedback loop back into PostgreSQL

### MAIN SESSION VIEW — session.html (most important page)

This is the primary page the clinician uses. It has three zones:

ZONE 1 — Top patient bar:
  Shows: patient name, MRN, age, session ID
  Shows: horizontal completeness progress bar (live, updates per turn)
  Shows: status badge ("Awaiting genomics", "Ready to grade", etc.)

ZONE 2 — Left/centre — U-Net metrics + chat window:
  A row of 4 metric cards showing:
    Whole tumor volume (cm³), necrosis ratio, edema ratio, midline shift
  Each card has a colored sub-label (warn/danger/ok) based on thresholds.
  
  Below the metric cards: the conversational chat window.
  Chat history scrolls upward. LLM messages appear as blue-tinted 
  bubbles on the left. User answers appear right-aligned.
  
  When response_type = "question":
    Render the question_text as a bold header inside the AI bubble.
    If question_type = "MCQ": render options as radio-style clickable divs.
    If question_type = "MSQ": render options as checkbox-style clickable divs.
    Options with type="free_text": reveal a text <input> on click.
    Options with type="skip": deselect all other options on click.
    A "Submit answer" button sends the selected option IDs + any 
    free_text to POST /chat as JSON.
  
  When response_type = "report":
    Render a structured report card inline in the chat:
      - WHO grade (large, colored pill: Grade 4 = red, Grade 3 = orange,
        Grade 2 = amber, Grade 1 = green)
      - Diagnosis label
      - Confidence score (shown as progress bar)
      - Survival category + estimated median OS
      - Favorable / unfavorable factors as pill tags
      - Treatment flags (info/warning/critical badges)
      - Recommendation summary paragraph
      - Disclaimer in small muted text
    A "Download report PDF" button (can be a stub for now).

ZONE 3 — Right panel (always visible):
  Section: "Completeness" — shows percentage + fields remaining count
  Section: "Key fields" — list of ~10 most important fields with 
    their current value or "not tested" in muted grey italic
  Section: "Auto-flags" — live list of triggered treatment flags
    (colour-coded: info = blue, warning = amber, critical = red)
  Section: "Provisional grade" — shows current best-guess grade

### BACKEND LOGIC — /chat endpoint (core of the app)

POST /chat
Request body: {
  "session_id": "uuid",
  "user_message": "string or null",
  "question_id": "string or null",        // question being answered
  "answer_ids": ["a", "c"],               // selected option IDs
  "answer_free_text": "string or null"    // if "Other" selected
}

Processing steps (in this exact order):
  1. Load session from PostgreSQL
  2. Save user message to messages table (role="user")
  3. If this is answering a question (question_id is not null):
     a. Look up fields_targeted from questions_asked
     b. Map selected answer IDs → field values using the option 
        clinical_flag values from the stored question JSON
     c. Update patient_context JSONB columns (symptoms, genomics, etc.)
     d. Update fields_populated map for each field
     e. Mark question as answered in questions_asked
  4. Call services/completeness.py → recompute completeness_score
     Store result in patient_context.completeness_score
  5. Call services/rag.py → embed the top 3 missing fields as a query
     Retrieve top 5 matching knowledge chunks
  6. Call services/context_builder.py → assemble full input_context
     Include: session_meta.questions_already_asked (list of 
     question_ids from questions_asked table) so Llama never re-asks
  7. Call services/llm.py → Llama returns JSON
  8. Validate JSON: must have response_type = "question" or "report"
  9. Save LLM response to messages table (role="assistant", 
     content_json = parsed JSON)
  10. If response_type = "question":
      Save to questions_asked (question_id, type, fields_targeted, 
      question JSON stored in content_json of the message)
  11. If response_type = "report":
      Save to diagnostic_reports table (all structured fields)
  12. Return the LLM JSON to frontend

### COMPLETENESS SCORING — services/completeness.py

Total fields tracked: ~45 (across symptoms, clinical, genomics, 
vasari, pathology, labs, treatment_history)

Weight tiers:
  CRITICAL fields (weight 3 each): 
    symptom_duration_days, seizure_onset_new, headache_pattern,
    age_at_diagnosis, idh_status, mgmt_methylation, karnofsky_score,
    enhancement_pattern, necrosis_on_histology, tert_promoter_mutation
  HIGH fields (weight 2 each):
    extent_of_resection, speech_deficit, motor_deficit, visual_deficit,
    cdkn2a_b_deletion, ependymal_invasion, mmse_score
  MEDIUM/LOW fields (weight 1 each): everything else

Score = sum(weight × 1 if field populated else 0) / sum(all weights)

This function is called after every /chat request. The result is 
stored in patient_context.completeness_score and passed as 
session_meta.completeness_score in every Llama call.

IMPORTANT: The LLM does NOT compute this. It only reads it.

### NOTES + CONSTRAINTS

1. The completeness_score must ONLY be computed by 
   services/completeness.py — never trust the LLM's own estimate.

2. The system prompt for Llama is provided in the attached 
   neuroassist_spec.html (System Prompt tab). Use it verbatim.

3. Always pass session_meta.questions_already_asked as a list of 
   question_id strings so Llama never re-asks the same question.

4. If Groq returns a response that fails JSON parsing, retry once 
   with a "respond ONLY with valid JSON, no preamble" prefix added 
   to the user message. If it fails again, return a safe error 
   response to the frontend.

5. If bevacizumab_use = true in patient_context, append a visible 
   banner in the chat window: "Anti-VEGF therapy detected — ET 
   volume interpretation may be affected."

6. The diagnostic report is assistive only. Always display the 
   disclaimer at the bottom of every report view:
   "NeuroAssist is a clinical decision support tool. All outputs 
   must be reviewed by a qualified neuro-oncologist before any 
   clinical action is taken."

7. Do NOT use Bootstrap or Tailwind. All CSS must be custom, 
   matching the hospital palette above. The UI mockup is attached 
   as a reference image.
